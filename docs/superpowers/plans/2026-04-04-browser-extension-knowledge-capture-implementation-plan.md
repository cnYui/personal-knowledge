# Browser Extension Knowledge Capture Implementation Plan

> Goal: 基于 `jumpAI` 扩展新增“选中文字后保存到个人知识库”的能力，并在 `personal-knowledge-base` 后端提供一个轻量保存入口，使用户能够在 AI 平台页面中完成选取、编辑、发送、保存成普通记忆的闭环。

## 1. 实施原则

1. 第一版只做“选中文字 -> 编辑 -> 保存为普通记忆”
2. 浏览器扩展只负责采集与发送，不负责知识图谱构建
3. 运行时边界保持清晰：
   - 插件是浏览器扩展
   - Web 前端是主产品 UI
   - 后端是统一存储与处理入口
4. 先打通最短闭环，再考虑批量保存、整条问答保存等增强功能
5. 尽量复用 `jumpAI` 已有平台注入与侧边面板能力，不重写新扩展

## 2. P0 总目标

P0 完成后，系统应满足：

- 用户在已支持 AI 平台页面中选中文字后，`jumpAI` 能稳定识别选区
- 插件侧边面板出现 `保存选中文本` 入口
- 用户可在插件面板中编辑标题与正文
- 插件可调用 `personal-knowledge-base` 后端接口保存内容
- 后端将内容保存成普通 memory
- 用户能在 `memories` 页面看到这条新记忆
- 保存过程不自动触发知识图谱写入

## 4. 后端阶段

### 阶段 1：新增浏览器采集保存接口

**目标：** 给浏览器扩展提供一个轻量、稳定的保存入口。

**任务：**

- 新增接口，例如：
  - `POST /api/memories/clip`
- 定义请求体 schema：
  - `title`
  - `content`
  - `source_platform`
  - `source_url`
  - `source_type`
- 返回统一保存结果：
  - 新记忆 ID
  - 标题
  - 创建时间

**行为要求：**

- 保存成功后只写入普通 memory
- 不自动调用图谱写入逻辑
- 不影响现有记忆保存主流程

**测试：**

- 合法请求可成功创建 memory
- 缺字段请求返回结构化错误
- 超长或空内容处理符合现有风格

**完成标准：**

- 后端有一个专门面向插件采集的轻量保存接口

### 阶段 2：复用 memory 存储模型

**目标：** 不引入第二套“采集内容”存储体系。

**任务：**

- 确认 memory 表已能容纳以下来源信息：
  - `source_platform`
  - `source_url`
  - `source_type = browser_clip`
- 如现有模型缺失字段，做最小补充
- 确保 `memories` 页面可以自然展示这些新增记忆

**测试：**

- 保存后的 memory 在记忆管理页查询结果中可见
- 不影响现有上传或手动新增记忆流程

**完成标准：**

- 插件采集内容与普通记忆统一收口

## 5. `jumpAI` 扩展阶段

### 阶段 3：实现选区监听

**目标：** 在 AI 页面中稳定识别用户当前选中的文字。

**任务：**

- 在 content script 中监听选区变化
- 提取选中文本
- 获取当前平台标识和当前页面 URL
- 过滤空选区、无效选区

**实现建议：**

- 不依赖消息气泡结构
- 第一版只围绕普通文本选区
- 尽量避免频繁渲染或抖动更新

**测试：**

- ChatGPT、Gemini、Kimi 等至少两个当前已支持平台上可稳定获取选中文本
- 清空选区后保存入口消失

**完成标准：**

- 扩展能拿到稳定的文本选区内容

### 阶段 4：侧边面板新增“保存模式”

**目标：** 在不重写 `jumpAI` 的前提下，为知识采集新增一套侧边面板状态。

**任务：**

- 保留当前导航模式
- 新增保存模式
- 当检测到选区时，面板中出现：
  - `保存选中文本`
- 点击后切换到保存模式

**保存模式字段：**

- 标题输入框
- 正文输入框
- 来源平台（只读）
- 来源 URL（只读）
- 保存按钮
- 返回导航模式按钮

**默认值：**

- 标题：选中文本前 20~30 个字
- 正文：选中文本全文

**测试：**

- 导航模式与保存模式切换正常
- 用户编辑内容不会被意外清空

**完成标准：**

- 用户可以在插件自己的 UI 中编辑后再保存

### 阶段 5：发送到个人知识库

**目标：** 打通扩展到后端的保存链路。

**任务：**

- 在 `jumpAI` 中实现请求发送逻辑
- 支持配置或固定本地后端地址
- 成功后给出明确提示：
  - `已保存到记忆管理`
- 失败时保留表单内容并提示错误

**错误处理：**

- 网络不可达
- 后端返回 4xx/5xx
- 跨域或权限问题

**测试：**

- 成功时返回的 memory 数据可用于面板提示
- 失败时用户输入不丢失

**完成标准：**

- 扩展采集内容可真实进入 `personal-knowledge-base`

## 6. 集成阶段

### 阶段 6：前后端联调

**目标：** 验证从 AI 页面选中文本到记忆管理页面出现新内容的完整闭环。

**联调流程：**

1. 打开 AI 平台页面
2. 选中一段文本
3. 在 `jumpAI` 面板中点击 `保存选中文本`
4. 编辑标题与正文
5. 点击保存
6. 在 `memories` 页面确认新记忆出现

**完成标准：**

- 用户不需要复制粘贴也能完成知识采集

## 7. P1 扩展顺序

P0 完成后，P1 建议按以下顺序推进：

1. 自动生成更好的默认标题
2. 保存成功后显示最近一次保存结果
3. 支持“保存当前整条问答”
4. 在插件中显示最近一次知识库连接状态
5. 如果后续准备统一仓库结构，再将 `jumpAI` 迁入 `browser-extension/`

## 8. 风险控制

### 风险 1：不同平台的页面选区行为不一致

控制：

- 第一版优先围绕普通文本选区
- 不依赖重 DOM 注入
- 先保证主要平台可用

### 风险 2：插件内新增表单后状态混乱

控制：

- 明确区分导航模式与保存模式
- 保存失败时不清空用户输入

### 风险 3：后端接口与现有 memory 流程冲突

控制：

- 复用 memory 主模型
- 不引入自动入图谱
- 只做轻量新入口

### 风险 4：项目组织继续分叉

控制：

- 第一版先逻辑联通
- 后续再考虑统一为同一代码库中的多应用结构

## 9. 完成定义

P0 算完成，必须同时满足：

- `jumpAI` 能检测选中文字
- 侧边面板能进入保存模式
- 用户可编辑标题与正文
- 保存请求能成功写入 `personal-knowledge-base`
- 内容在 `memories` 页面可见
- 整条链路不自动触发知识图谱写入
- 插件依然只是采集入口，不承担图谱构建职责
