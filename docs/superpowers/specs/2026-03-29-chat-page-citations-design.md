# Chat 页面重写与引用展示设计

## 1. 背景与目标

当前 `/chat` 页面已经具备基础对话能力，但存在三个问题：

1. 页面头部仍保留“知识库对话 / 直接提问，让 AI 基于你的学习记忆进行回答。”这类说明性文案，不符合当前更聚焦的聊天工作区体验。
2. 在聊天记录较长、用户上下滚动查看上下文时，顶部“个人知识库”栏和左侧导航不会保持稳定的固定位置，影响持续浏览体验。
3. AI 回复虽然已经从后端流式接收 `references`，但前端没有把引用编号、底部出处和悬浮内容展示出来，用户无法判断回答使用了哪些知识内容。

本次目标是在不要求后端提供精确句级锚点的前提下，以前端优先方案完成一个可用、可持久化的引用型聊天界面。

## 2. 范围与非目标

### 2.1 本期范围

- 重写 `/chat` 页面内容区结构，移除说明性文案。
- 调整整体布局，使左侧导航和顶部栏在浏览聊天上下文时保持固定。
- 让聊天消息区成为主要滚动容器。
- 为 assistant 消息增加 `references` 持久化存储。
- 将 AI 回复渲染为“正文带编号引用 + 底部小字号出处列表 + 悬浮查看原文”。
- 兼容流式输出中的内容增量更新与引用更新。

### 2.2 非目标

- 不改造后端流式协议为精确句级引用坐标。
- 不引入新的富文本编辑器或复杂标注系统。
- 不实现跨消息聚合的全局引用面板。
- 不改造非聊天页面的布局和视觉语言。

## 3. 核心设计决策

### 3.1 引用方案

采用用户确认的“方式 2：消息级持久化引用”：

- `ChatMessage` 扩展 `references?: ChatReference[]` 字段。
- assistant 消息在流式阶段接收到 `references` 后，立即写入当前消息对象并保存到 `localStorage`。
- 页面刷新后，历史消息仍可显示引用编号与出处。

### 3.2 句内编号策略

由于后端当前仅返回引用集合，不返回逐句锚点，本期采用前端近似映射：

- assistant 回复按可见句子切分。
- 当消息存在引用时，将引用编号集中追加到一个或多个句尾，以保证用户能在正文中看到编号入口。
- 底部统一展示编号与出处原文。

为了避免破坏 Markdown 结构，本期优先使用保守策略：

- 对纯段落文本进行句尾编号追加。
- 若内容呈现为复杂 Markdown 结构，则仍保证底部出处列表完整可见，正文编号以尽量不破坏渲染为前提。

### 3.3 布局与滚动策略

采用“应用骨架固定，内容区内部滚动”的结构：

- `AppLayout` 使用 `height: 100vh`，限制整个应用在首屏高度内。
- `SideNav` 在左侧全高展示，不参与主内容滚动。
- `TopBar` 采用 sticky 顶部固定。
- `main` 容器禁用页面级滚动，由聊天页面内部控制消息区滚动。
- `/chat` 页面内部采用三段式布局：
  - 顶部操作区
  - 中部消息滚动区
  - 底部输入区

## 4. 组件与模块调整

### 4.1 布局层

涉及文件：

- `frontend/src/components/layout/AppLayout.tsx`
- `frontend/src/components/layout/TopBar.tsx`
- `frontend/src/components/layout/SideNav.tsx`

调整方向：

- `AppLayout` 改为固定视口高度，并让右侧内容区具备独立滚动管理。
- `TopBar` 改为 sticky，补充层级和背景，避免滚动时内容穿透。
- `SideNav` 设置固定宽度和全高，必要时自身可独立滚动，但不随聊天内容滚动。

### 4.2 Chat 页面层

涉及文件：

- `frontend/src/pages/KnowledgeChatPage.tsx`

调整方向：

- 去掉 `PageHeader` 中的标题和描述文案展示方式，改为更轻量的操作头。
- 保留“清空对话”按钮。
- 重新组织空态、消息列表和输入框，使输入框稳定贴近底部，消息区单独滚动。

### 4.3 消息渲染层

涉及文件：

- `frontend/src/components/chat/ChatMessageList.tsx`
- 可能新增一个局部引用渲染辅助函数或内联子组件（仍放在同文件或 chat 组件目录下）。

调整方向：

- assistant 消息渲染时支持带编号的正文展示。
- 在消息底部显示小字号引用列表。
- 引用编号支持鼠标悬浮 `Tooltip` 或等价弹层，显示原始语料/知识图谱内容。
- 流式消息在输出过程中也能显示当前已接收的引用。

### 4.4 数据与状态层

涉及文件：

- `frontend/src/types/chat.ts`
- `frontend/src/hooks/useChat.ts`
- `frontend/src/services/chatApi.ts`

调整方向：

- `ChatMessage` 增加 `references` 字段。
- 在 `sendChatMessageStream` 中，当收到 `references` 事件时，更新当前 assistant 消息并写回 `localStorage`。
- `useSendChatMessage` 将 `references` 继续暴露给页面/消息列表，用于流式渲染期间展示。

## 5. 数据流设计

### 5.1 发送消息

1. 用户发送文本。
2. 前端先写入一条 user 消息到 `localStorage`。
3. 前端创建一条 assistant 占位消息，初始 `content` 为空、`references` 为空数组或省略。
4. 页面立即刷新出会话记录。

### 5.2 流式回复

1. 后端返回 `content` 事件时，逐步拼接 assistant 的 `content`。
2. 后端返回 `references` 事件时，更新 assistant 消息的 `references`。
3. 每次内容或引用变化都写回 `localStorage`。
4. UI 在流式阶段同时使用 `streamingContent` 和 `references` 渲染临时消息。

### 5.3 完成与刷新

1. 流式完成后保留持久化消息。
2. 通过 react-query 失效 `chat-messages` 查询，重新读取统一数据源。
3. 历史消息中已带完整 `references`，无需额外恢复逻辑。

## 6. 引用内容展示规则

每个引用项统一生成顺序编号：`[1]`、`[2]`、`[3]`。

出处内容优先级：

- 若存在 `fact`，优先展示 `fact`
- 否则展示 `summary`
- 否则展示 `name`
- 如以上都缺失，则回退为类型说明（例如 `entity` / `relationship`）

展示形式：

- 正文中的编号使用较小字号和可交互样式。
- 底部出处区使用小字号、弱化色，按编号逐项列出。
- 悬浮时显示完整内容，避免底部摘要过长影响主阅读流。

## 7. 错误处理与兼容性

- 若后端未返回 `references`，assistant 消息按普通回答渲染，不展示引用区。
- 若 `references` 返回为空数组，正文不追加编号，底部不展示出处列表。
- 若流式中途失败，保留已生成内容，并按现有错误逻辑显示错误状态。
- 若旧的 `localStorage` 历史消息没有 `references` 字段，前端按可选字段兼容处理。

## 8. 验收标准

满足以下条件即视为完成：

1. `/chat` 页面不再显示“知识库对话”与“直接提问，让 AI 基于你的学习记忆进行回答。”文案。
2. 浏览长聊天记录时，顶部“个人知识库”和左侧导航保持固定，不随消息内容滚动离开视口。
3. assistant 消息在存在引用时，正文中可以看到编号引用入口。
4. assistant 消息底部以小字号展示编号对应的原始语料或知识图谱内容。
5. 鼠标悬浮正文中的引用编号时，会弹出对应的引用内容。
6. 刷新页面后，历史 assistant 消息仍能显示引用。
7. 无引用消息、旧历史消息、流式进行中消息都能正常显示，不报错。

## 9. 实施顺序

1. 扩展聊天类型定义，支持消息级 `references`。
2. 改造 `chatApi` 与 `useChat`，持久化引用数据并保留流式展示能力。
3. 重构 `AppLayout` / `TopBar` / `SideNav`，实现固定骨架布局。
4. 重构 `KnowledgeChatPage`，建立操作区、消息区、输入区三段式结构。
5. 改造 `ChatMessageList`，实现正文编号、底部出处、悬浮弹窗。
6. 回归检查空态、错误态、滚动体验和刷新后的历史消息展示。
