# RAGFlow 中的 Agentic RAG 实现详解

## 1. 文档目的

本文档用于系统梳理 RAGFlow 项目中 Agentic RAG 的实现方式，重点回答以下问题：

- 什么是这里的 Agentic RAG
- 为什么项目里要引入 Canvas
- Retrieval 为什么要被包装成工具
- Agent 是如何自己决定何时检索、检索几轮的
- `Canvas.retrieval` 和 citation 的作用分别是什么
- 传统 RAG 和 Agentic RAG 在 RAGFlow 中的差异是什么

本文基于当前项目代码结构进行说明，核心目录包括：

- `api/`
- `agent/`
- `rag/`

---

## 2. 先说结论：这里的 Agentic RAG 是什么

RAGFlow 中的 Agentic RAG，不是简单的“先检索，再生成”固定流水线，而是：

1. 使用 `Canvas` 作为整个工作流运行容器
2. 使用 `Agent` 作为具备推理和工具调用能力的核心节点
3. 将 `Retrieval`、MCP 工具、其他能力节点统一包装成 LLM 可调用的工具
4. 允许模型在多轮推理中自己决定：
   - 要不要检索
   - 什么时候检索
   - 检索几轮
   - 检索后是否继续调用其他工具
   - 什么时候结束并产出最终答案

所以，这里的 Agentic RAG 本质上是：

**让 RAG 从固定流程，升级为可由 Agent 自主调度的能力。**

---

## 3. 为什么这里要有 Canvas，它不是一个 RAG 吗

这是理解整个设计的关键。

### 3.1 RAG 只是能力，不是完整运行时

一个最简单的 RAG 系统通常只有以下逻辑：

1. 接收用户问题
2. 去知识库检索相关片段
3. 把片段拼进 Prompt
4. 调用模型回答

这种实现可以工作，但它只覆盖了“检索 + 生成”本身。

而一个真正面向产品的系统，还需要处理很多额外问题：

- 多轮会话历史
- 用户输入参数和变量
- 文件上传与解析
- 节点间数据传递
- 分支、循环、条件判断
- 工具调用
- 流式输出
- 中断取消
- 调试与追踪
- 会话恢复
- 引用和证据聚合

这些内容已经超出了传统 RAG 的范畴。

### 3.2 Canvas 是工作流容器和运行时状态机

在 RAGFlow 里，`Canvas` 的职责不是“替代 RAG”，而是为 RAG 和 Agent 提供运行环境。

它负责：

- 加载 DSL 图定义
- 初始化本次任务的组件实例
- 保存 `globals`、`history`、`retrieval`、`memory`
- 按路径调度组件执行
- 处理节点输出与下游跳转
- 输出工作流生命周期事件

可以把它理解为：

- `Canvas` = 工作流引擎 / 任务容器 / 状态机
- `Retrieval` = 检索能力
- `LLM` = 生成能力
- `Agent` = 具备推理和工具调用能力的高级节点

所以它当然还是一个 RAG 系统，只不过是一个**可编排、可追踪、可扩展的 RAG 平台**。

---

## 4. RAGFlow 中的两种 RAG 形态

### 4.1 传统 RAG

传统 RAG 在这里通常长这样：

`Begin -> Retrieval -> LLM -> Message`

特点：

- 流程固定
- 每次都会检索
- 检索通常就是一轮
- 行为可预测，稳定性高
- 更适合知识库问答等高确定性场景

### 4.2 Agentic RAG

Agentic RAG 通常长这样：

`Begin -> Agent -> Message`

但这个 `Agent` 内部挂了 `Retrieval` 工具。

特点：

- 外层流程看起来更短
- 检索不是固定节点，而是 Agent 的一种能力
- 模型可以自己决定要不要调 `Retrieval`
- 可以多轮检索
- 可以和其他工具组合使用
- 灵活性更高，但也存在漏调工具的风险

---

## 5. 为什么要把 Retrieval 打包成工具

### 5.1 如果不打包成工具，会发生什么

如果 `Retrieval` 只是固定节点，那么流程就是写死的：

`Begin -> Retrieval -> LLM -> Message`

这种模式有明显限制：

- 每次都必须先检索
- 模型不能根据上下文决定是否检索
- 模型不能在检索后决定“还不够，再检索一次”
- 模型无法把知识库检索和网页检索、SQL、代码执行等能力统一调度

### 5.2 打包成工具后的收益

把 `Retrieval` 包装成 tool 后，Agent 可以像调用函数一样使用它。

这带来四个关键变化：

1. 检索变成按需发生
   - 不是固定一定要跑
   - 模型可以判断当前问题是否需要查知识库

2. 检索可以多轮发生
   - 第一轮没找到足够证据时，可以改写 query 再搜一轮

3. 检索和其他能力变成平级工具
   - 可以与网页搜索、MCP、SQL、其他 Agent 一起配合

4. 工作流更通用
   - Canvas 不需要为每种能力写死流程
   - 只要统一抽象成工具，就能被 Agent 调度

### 5.3 代价：会出现漏调工具的问题

会，而且这是 Agentic RAG 的典型风险。

因为一旦 Retrieval 被包装成 tool，是否调用它不再由代码强制，而是由：

- 模型能力
- 系统提示词
- tool 描述
- 当前上下文

共同决定。

于是会出现：

- 该检索却没检索
- 检索一轮后过早停止
- query 改写得不好，导致召回不足

因此：

- 想要更稳，适合用固定 RAG 链
- 想要更灵活，适合用 Agentic RAG
- 想要降低漏调风险，需要在 prompt 中显式要求必须使用 Retrieval

## 6. Retrieval 是如何被“打包成工具”的

这里不是只有 MCP 才能成为工具。

RAGFlow 里有两种工具来源：

### 6.1 本地类包装成工具

`Retrieval` 本质上是项目内部的一个 Python 类。

它通过参数类提供工具元信息，例如：

- 名称
- 描述
- 参数 schema
- 必填项

然后 `Agent` 会在初始化时：

1. 读取 DSL 中配置的 `tools`
2. 实例化这些工具对象
3. 调用 `get_meta()` 获取 function schema
4. 将这些 schema 注册到底层 LLM 的 tool calling 中

也就是说，本地工具的本质是：

**Python 类 + 参数元信息 + 调用包装器**

### 6.2 MCP 工具接入

如果配置的是 MCP server，那么系统会：

1. 读取 MCP server 的工具元数据
2. 转换为统一的 function tool schema
3. 通过 `MCPToolCallSession` 接入

所以 MCP 是工具来源之一，但不是唯一来源。

### 6.3 统一抽象层

在 LLM 看来，不管工具来自哪里，最终都会被统一成“可调用函数”：

- 本地 Retrieval -> function tool
- 本地 SQL / Crawler -> function tool
- MCP server tool -> function tool

因此，这里的工具系统本质上是统一抽象，而不是 MCP 专属机制。

## 7. Agent 是如何自己决定“何时检索、检索几轮”的

这是 Agentic RAG 最核心的地方。

### 7.1 不是代码写死了“检索几次”

代码并没有规定：

- 第一步必须检索
- 总共检索两轮
- 检索完成后一定直接回答

真正的机制是：

**LLM 在一个工具调用循环中自己决定下一步动作。**

### 7.2 决策依据是什么

模型每一轮会综合这些信息来做判断：

- 系统提示词
- 当前用户问题
- 多轮对话历史
- 上一轮工具执行结果
- 每个工具的名字、描述和参数定义

举例：

如果系统提示词写着：

- 每个问题必须先用 Retrieval
- 当证据不足时继续搜索
- 只有拿到知识库证据后才能回答

那么模型就更容易优先调用 Retrieval。

### 7.3 一轮决策流程长什么样

每一轮大体会经历：

1. 模型读取当前 history
2. 判断是否需要外部信息
3. 如果需要，则输出 `tool_calls`
4. 平台执行工具
5. 将工具结果追加回 history
6. 模型重新思考下一步
7. 直到模型认为“现在可以回答”

### 7.4 “几轮”是如何收敛的

收敛有两种方式：

1. 模型主动结束
   - 它判断当前证据足够，不再发起 tool call，直接回答

2. 平台上限终止
   - 由 `max_rounds` 设定最大轮数
   - 如果超过上限还没有停止，就终止循环

所以：

- “要不要再搜一轮”主要由模型决定
- “最多允许搜多少轮”由平台决定

这就是 Agentic RAG 的典型结构：

**模型负责策略，平台负责边界。**

## 8. `Canvas.retrieval` 是什么

`Canvas.retrieval` 可以理解成：

**当前这次工作流运行中累积出来的证据池。**

每次 Retrieval 完成后，命中的：

- chunks
- 文档聚合信息 `doc_aggs`

都会写入 `Canvas.retrieval`。

它的作用不是立即显示给用户，而是作为本次运行共享的“证据缓存层”。

### 8.1 为什么不只把结果放在当前工具输出里

如果只放在当前工具输出里，会有几个问题：

- 后续节点不好统一访问
- 多次检索结果难以汇总
- 最终回答后的 citation 很难统一生成

所以系统把检索结果额外写入全局引用池，方便后续统一使用。

### 8.2 多轮检索时尤其重要

如果 Agent 检索了多轮：

- 第 1 轮找到 A、B
- 第 2 轮找到 C、D

这些结果都可以累计进 `Canvas.retrieval`。

最终答案和 citation 阶段可以统一从这里取证据，而不是只看最后一次工具返回值。

---

## 9. “根据 Canvas.retrieval 生成 citation”是什么意思

这句话的意思是：

1. 先用 Retrieval 找到证据
2. 把证据保存到 `Canvas.retrieval`
3. Agent 先生成一版答案
4. 然后再读取 `Canvas.retrieval` 中的证据
5. 再让模型给答案补上引用标记或出处说明

### 9.1 为什么要分成两步

因为“生成回答”和“组织引用”是两个不同任务：

- 回答生成关注内容质量
- citation 生成关注证据绑定

把它们分开有几个好处：

1. 逻辑更清晰
   - 先回答，再补引用

2. 支持统一引用
   - 不管检索发生了几轮，都从统一证据池拿材料

3. 前端更容易展示
   - 能附带 chunk、文档名、URL 等信息

### 9.2 citation 的作用

citation 的主要作用是：

- 给答案提供出处
- 提高可解释性
- 便于用户核查证据
- 支持前端展示参考文档和命中片段

所以你可以把三者关系理解为：

- `Retrieval`：找材料
- `Canvas.retrieval`：存材料
- `citation`：把材料挂到答案上

## 10. 整体运行流程总结

下面用概念流程总结整个 Agentic RAG：

1. 用户问题进入 `/canvas/completion`
2. 后端读取 DSL，创建 `Canvas`
3. `Canvas` 初始化当前任务上下文
4. 工作流进入 `Begin -> Agent`
5. Agent 将 Retrieval 和其他能力注册为 tools
6. 模型开始工具调用循环
7. 当模型认为需要知识库证据时，调用 Retrieval tool
8. Retrieval 执行检索，并把结果写入 `Canvas.retrieval`
9. 工具结果回填到对话历史
10. 模型继续判断是否要再搜一轮
11. 当模型认为证据足够时，产出答案
12. 若开启 citation，则根据 `Canvas.retrieval` 统一补引用
13. Message 节点将结果流式输出给前端

---

## 11. 什么时候该用传统 RAG，什么时候该用 Agentic RAG

### 更适合传统 RAG 的场景

- 知识库问答
- 企业内部文档检索
- 希望结果稳定可控
- 不希望出现漏调工具
- 成本敏感，想减少多轮推理

### 更适合 Agentic RAG 的场景

- 复杂任务拆解
- 多工具协同
- 需要动态判断是否检索
- 需要多轮搜索与迭代
- 希望知识库检索与外部搜索联合使用

### 一句话选择建议

- 想要稳：用固定 RAG
- 想要活：用 Agentic RAG

---

## 12. 最后的理解框架

如果要用一句话概括 RAGFlow 里的设计：

**Canvas 提供工作流运行时，Retrieval 提供检索能力，Agent 负责自主决策，三者结合构成 Agentic RAG。**

也可以进一步压缩成三层：

- Canvas：负责运行
- Retrieval：负责找证据
- Agent：负责决定下一步

这也是为什么这个项目看起来不像一个“纯 RAG 脚本”，而像一个“可编排的 Agentic RAG 平台”。

---

## 13. 附：一句话回答常见疑问

### 为什么要有 Canvas
因为产品级 RAG 需要状态、调度、变量、分支、流式输出和追踪，不能只有检索和生成。

### 为什么要把 Retrieval 变成工具
因为这样 Agent 才能自主决定何时检索、检索几轮、是否结合其他工具一起工作。

### 会不会漏掉 Retrieval
会，所以高确定性场景更适合固定 RAG 链路；Agentic RAG 需要 prompt 约束和合理的 `max_rounds`。

### Retrieval 是 MCP 吗
不一定。它可以是本地类包装成 tool，也可以是 MCP server 暴露出来的工具，最终都会被统一成 function tool。

### citation 的本质是什么
把检索得到的证据池挂接到最终答案上，使答案可追溯、可验证。
